我更喜欢把后端工程化理解成一组让代码长期保持可改的约束：模块边界清楚，接口语义稳定，测试覆盖关键行为，运行时问题能够被观察到。

## 目录不是装饰

Go 项目里，目录结构应该服务于业务边界，而不是为了显得复杂。小项目可以保持扁平，等到重复出现再抽象。真正值得优先处理的是包之间的依赖方向。

- 把核心领域逻辑和外部适配层分开。
- 让配置、日志、存储、网络等基础设施可替换。
- 对外暴露稳定接口，对内保留重构空间。

## 测试要覆盖行为

我会优先写能锁住行为的测试，而不是机械追求覆盖率数字。尤其是网关、RPC、事务这类系统，边界条件和失败路径通常比快乐路径更值钱。

```go
func TestRegistryInitFailsFast(t *testing.T) {
    err := InitRegistry("unknown")
    require.Error(t, err)
}
```

## 记录运行现场

每次修复都应该留下三个信息：复现路径、判断依据和验证命令。下一次遇到相似问题时，这些记录比模糊的“经验”更可靠。
